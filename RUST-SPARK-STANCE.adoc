== Summary

This repository's application logic is written in Rust (Bevy game engine
+ optional Z3). Per the estate language policy
(`standards/rhodium-standard-repositories/spec/LANGUAGE-POLICY.adoc`
§Terminology), *"Rust" means "Rust/SPARK"*: Rust is the primary
implementation language now, but the project is *designed to admit
SPARK/Ada verification modules* across a stable, Idris2-typed boundary,
with Zig as the FFI layer. Rust here is never the ABI/API/FFI layer.

This document is the project's compliance stance against that policy.
Before this audit (2026-05-18) the crate was bare Rust with no seam, no
SPARK admission path, and no stance doc — non-compliant. The seam below
brings it into structural compliance; the remaining proof work is
tracked in `PROOF-NEEDS.md`.

== What is consensus-/correctness-critical here

This is a single-player + leaderboard logic-puzzle game, *not* a
distributed ledger. There is no nonce search, no difficulty-retarget
loop, and no chain of blocks. The audit's blockchain-style invariant
names map onto the real correctness-critical surface as follows:

[cols="1,2,3",options="header"]
|===
| Audit framing | Real counterpart | Where in code

| Nonce search soundness
| Verification soundness — no false positives
| `src/verification/mod.rs::verify_level_solution`,
  `src/verification/z3_integration.rs::verify_formula`

| Hash collision-resistance assumption
| Submission-signature binding (SHA-256 keyed hash)
| `src/network/client.rs::sign_proof`

| Difficulty target monotonicity
| Pack/level difficulty in range + non-decreasing progression
| `src/levels/mod.rs` (`LevelPack::difficulty`, `next_level`)

| Chain-extension validity
| Board well-formedness preserved under placement/move
| `src/game/board.rs::place_piece` / `move_piece`,
  `src/game/validation.rs`
|===

== The seam

[cols="1,3",options="header"]
|===
| File | Role

| `src/abi/ProofOfWork/ABI/Types.idr`
| Idris2 boundary types — faithful total models of `LogicPiece`,
  `BoardState`, `Level`, `GoalCondition`, `ExportedProof`. Constructor
  order is ABI-significant and matches the Rust enums.

| `src/abi/ProofOfWork/ABI/Invariants.idr`
| Type-level statements of invariants I1–I7. **I5 is a DISCHARGED,
  machine-checked proof** (`decNonDecreasing` + `builtinPackMonotone`).
  **I6 is an explicit cryptographic-hardness ASSUMPTION**
  (`sha256CollisionResistant`). I1–I4 and I7 are stated as erased
  **OWED** obligations the Rust does not yet establish. Authoritative
  per-ID status: `PROOF-NEEDS.md` § Invariant register.

| `src/abi/ProofOfWork/ABI/Foreign.idr`
| Zig-FFI boundary: u32/u8 range obligations for marshalling, the stable
  C-ABI `VerifyResult` codes, and the contract tying a `VerifyAccepted`
  return to an I1 certificate.

| `src/abi/proof-of-work-abi.ipkg`
| Idris2 package mapping the `ProofOfWork.ABI.*` namespace to the source
  tree. Before 2026-05-19 the modules sat flat in `src/abi/` and the seam
  **did not build** (`import ProofOfWork.ABI.Types` failed) — present but
  never machine-checked. This package makes it actually verifiable.
|===

The seam is now machine-checked: `idris2 --build proof-of-work-abi.ipkg`
(run from `src/abi/`) typechecks all three modules, and
`.github/workflows/abi-verify.yml` enforces it on every change to
`src/abi/**` (estate-standard `snazzybucket/idris2` container, mirroring
the stapeln cerro-torre Idris2 gate). The discharged proofs therefore
cannot silently regress, and an OWED postulate cannot quietly become a
`believe_me`.

== SPARK/Ada admission path

The boundary is the Idris2 seam above. A future SPARK/Ada
re-implementation of the verifier (the highest-value target — see
`Invariants.idr` I1/I2) sits on the far side of
`Foreign.acceptedImpliesVerified` without changing the contract: it must
return `VerifyAccepted` only when it can produce a `VerifiedSolution`
certificate. No SPARK code exists yet; the seam exists so it *can* be
added without redesign, which is exactly what §Terminology requires of a
"designed to admit SPARK/Ada" Rust project.

== Honest gaps

* The non-Z3 (`#[cfg(not(feature="z3-verify"))]`) verifier accepts on
  connectivity alone with *no* SMT step (`src/verification/mod.rs`). This
  is a real soundness defect (Invariant I2, postulated/owed), not just a
  missing proof.
* `verify_formula` / `validate_proof_locally` in
  `src/verification/z3_integration.rs` do not parse the level's actual
  theorem — they assert a hardcoded `P,Q ⇒ R` shape. Soundness for
  arbitrary levels is unestablished.
* `sign_proof` is a keyed SHA-256 hash, not a MAC/signature scheme;
  integrity is conditional on SHA-256 hardness (I6) and the construction
  has known keyed-hash caveats out of seam scope.
* The repo's own `.claude/CLAUDE.md` language table (ReScript/Tauri/Deno)
  is stale estate drift and is superseded by the estate policy and this
  stance doc; it should be reconciled separately.
